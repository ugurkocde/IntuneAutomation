Every script page on this site has a **Deploy to Azure** button. This walkthrough shows exactly what happens after you click it, with a screenshot at every step of a real deployment: the [Stale Device Cleanup Alert](/script/stale-device-cleanup-alert/) runbook, deployed into a lab tenant, granted its permissions, and run to a successful job with zero errors. Nothing here is mocked up; even the troubleshooting section comes from an error this very deployment hit.

If you want the conceptual background on why managed identities beat app registrations for unattended scripts, read the [managed identity fundamentals guide](/blog/run-intune-script-azure-automation-managed-identity/) first. This post is the hands-on companion: click, screenshot, click.

## Before you start

You need three things:

- An Azure subscription where you can create resource groups and Automation accounts
- A role that can grant admin consent for Graph application permissions (Global Administrator or Privileged Role Administrator). This is the one step an Intune Administrator role alone cannot do, because granting app roles to an identity is a directory operation
- For notification scripts, a licensed mailbox to use as the sender

Azure Automation includes 500 free job minutes per month. A weekly notification runbook uses a fraction of that.

## Step 4: Grant the Graph permissions (the step with no portal UI)

Here is the part everyone gets stuck on: **you cannot grant Graph application permissions to a managed identity through the portal**. There is no App registrations entry for it and no consent button. The grant is an app role assignment created through the Graph API, and the fastest place to do that is Cloud Shell, because it already runs with your admin sign-in.

Open Cloud Shell from the portal header and run this, with your identity name and your script's permission list:

```bash
IDENTITY_NAME="aa-intuneautomation-demo"
PERMISSIONS="DeviceManagementManagedDevices.Read.All Mail.Send"

GRAPH_SP_ID=$(az ad sp show --id 00000003-0000-0000-c000-000000000000 --query id -o tsv)
MSI_ID=$(az ad sp list --display-name "$IDENTITY_NAME" --query "[0].id" -o tsv)

for PERM in $PERMISSIONS; do
  ROLE_ID=$(az ad sp show --id 00000003-0000-0000-c000-000000000000 \
    --query "appRoles[?value=='$PERM'].id" -o tsv)
  az rest --method POST \
    --url "https://graph.microsoft.com/v1.0/servicePrincipals/$MSI_ID/appRoleAssignments" \
    --headers "Content-Type=application/json" \
    --body "{\"principalId\":\"$MSI_ID\",\"resourceId\":\"$GRAPH_SP_ID\",\"appRoleId\":\"$ROLE_ID\"}"
done
```

Each grant echoes back a JSON confirmation with your identity as the principal and Microsoft Graph as the resource:

![Cloud Shell returning the created app role assignment](/guides/runbook-managed-identity/08-cloudshell-grant.png)

Prefer PowerShell? The repository ships [grant-permissions-managed-identity.ps1](https://github.com/ugurkocde/IntuneAutomation/blob/main/grant-permissions-managed-identity.ps1), which wraps the same API calls.

To verify, open **Entra ID > Enterprise applications**, clear the application type filter, search for the Automation account name, and check **Security > Permissions**. Both scopes show as Application permissions with admin consent:

![Enterprise application Permissions blade showing both granted scopes](/guides/runbook-managed-identity/10-enterprise-app-permissions.png)

Grant only what the script page lists. If tenant policy considers application-level `Mail.Send` too broad, scope it to the sender mailbox with an Exchange Online application access policy.

## Step 5: Import Microsoft.Graph.Authentication

The scripts need a single module. Portal path: **Shared Resources > Modules > Add a module > Browse from gallery**, import `Microsoft.Graph.Authentication` for the PowerShell 5.1 runtime. Or from the same Cloud Shell:

```powershell
New-AzAutomationModule -AutomationAccountName "aa-intuneautomation-demo" `
  -ResourceGroupName "rg-intuneautomation-runbooks" `
  -Name "Microsoft.Graph.Authentication" `
  -ContentLinkUri "https://www.powershellgallery.com/api/v2/package/Microsoft.Graph.Authentication"
```

Wait for `ProvisioningState: Succeeded` (a few minutes) before running anything:

![Module import completed](/guides/runbook-managed-identity/11-module-import-succeeded.png)

## Step 7: Put it on a schedule

In the runbook, go to **Resources > Schedules > Add a schedule**, create a weekly recurrence, and bind the same parameter values. That is the finish line: the script now runs unattended forever, and revoking it later is as simple as deleting the resource group.

## Troubleshooting from the real deployment

**Quota error on deployment.** This walkthrough's first deployment failed with "exceeded your quota for Automation accounts in your subscription in the region". Subscriptions have a per-region Automation account quota. Fix: deploy into an existing account, or change the **Location** parameter to another region and redeploy, which is exactly what happened here:

![The quota error hit during this deployment](/guides/runbook-managed-identity/16-troubleshooting-quota-error.png)

**"Module 'Microsoft.Graph.Authentication' is not available in this Azure Automation Account."** The import from step 5 has not finished, or targeted a different runtime. Wait for Succeeded on the 5.1 runtime.

**403 or "Insufficient privileges" from Graph in the job.** A scope from the script page is missing on the identity, or the grant is still propagating. Check the Enterprise application's Permissions blade first, then give it a few minutes; freshly granted app roles do not always appear in the very next token.

**Mail send fails.** The `SenderUPN` mailbox must exist and be licensed, and the identity needs `Mail.Send`. App-only identities have no `/me`, which is why the scripts send through `/users/{SenderUPN}/sendMail`.
